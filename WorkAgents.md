# Agent 初始化协作模板

本文档用于初始化新项目的 `AGENTS.md`。使用时请复制本模板到目标项目，并把占位内容替换为该项目的真实约定。

## 通用规则
1. 始终使用中文思考/回复。
2. 不要在代码、示例配置或文档中硬编码 API Key、密码、Bearer Token 等敏感信息；新增配置优先从环境变量读取。
3. 不确定需求意图时，先询问再动手。
4. 大范围重构前先说明方案，等确认后再执行。
5. 不要回滚、覆盖或清理用户/其他 agent 的未提交改动，除非用户明确要求。

## 代码修改流程
1. 修改前先阅读相关代码和文档，不要凭空假设项目结构。
2. 修改完成后按项目要求执行最小必要验证。
3. 如果本次修改、探索或新增文档产生了会影响后续协作的长期约定，必须同步回归更新 `AGENTS.md`。
4. 判断是否需要更新 `AGENTS.md`：如果下次新 agent 不知道这条信息会明显走错目录、跑错命令、使用错误架构或提交敏感信息，则更新。
5. 如果只是当前 bug 原因、一次性排查记录、普通业务实现细节，则不要写入 `AGENTS.md`。
6. 不确定是否需要更新时，在最终回复中提示可能需要更新并说明原因。

## git 管理规则
1. 只有用户明确要求或项目规则要求时才提交 git commit。
2. 提交前必须查看 `git status`、`git diff` 和最近提交记录，确认只暂存本次任务相关文件。
3. 不要提交 `.env`、真实密钥、运行时数据、日志、构建缓存或无关文件。
4. commit message 使用中文，建议格式：`<类型>: <简要描述>`。
5. 常用类型：`功能`、`修复`、`重构`、`文档`、`配置`、`测试`。

## 工作区说明
1. 项目文档优先从以下路径读取：
   - `README.md`
   - `docs/*.md`
   - `<按项目补充其他核心文档>`
2. 项目文档使用中文，新增长期文档也优先使用中文。
3. 在这里说明项目的主要目录边界，例如：
   - `backend/`：后端服务
   - `frontend/`：前端应用
   - `scripts/`：开发、构建或数据脚本
4. 依赖变更后必须在对应子项目执行同步命令，例如：
   - Python/uv：`uv sync`
   - Node/npm：`npm install`
   - 其他包管理器：`<按项目填写>`
5. 维护 `.env.example` 作为环境变量模板，不包含真实敏感值；新增环境变量时同步更新模板。
6. 如需项目变更记录，可以查看 git 提交历史。

## 代码风格
### 通用
- 保持最小正确修改，不做无关重构。
- 新增代码应遵循现有命名、目录结构和错误处理风格。
- 日志使用项目既有日志方案，不使用临时 `print`/`console.log` 调试后遗留。

### Python
- 所有函数、方法添加 type hints。
- 使用 Pydantic model 定义 API 请求/响应结构（如项目使用 Pydantic）。
- 使用 `logging` 记录日志。

## 错误处理
- API 路由返回明确状态码和错误信息。
- 数据库、网络、文件系统等外部操作使用 try/except 或项目统一错误机制，并记录日志。
- 前端请求错误不要静默吞掉，应按项目约定展示或记录。

## 测试与验证
1. 新增 API 端点时同步补充测试。
2. 数据库 schema 变更必须包含迁移脚本，并同步更新数据模型文档。
3. 功能实现后执行最小必要验证，常用命令按项目填写：
   - 后端：`<例如 uv run pytest && uv run ruff check .>`
   - 前端：`<例如 npm run build>`
   - 其他子项目：`<按项目填写>`
4. 如果验证无法执行，在最终回复中说明原因和风险。

## 安全与配置
- `.env`、本地数据库、日志、上传文件、构建缓存等运行时产物默认不提交。
- `.env.example`、示例 MCP/插件配置、文档中的 token 必须使用占位值。
- 新增第三方服务配置时优先支持环境变量，并在文档中说明用途和默认值。

## 数据库
- 使用迁移工具管理 schema 变更。
- 每次 schema 变更必须生成迁移脚本并提交。
- 在项目数据模型文档中维护表结构、字段含义和重要约束。
